# Internationalization (i18n)

Every user-facing string goes through localization. No hardcoded Turkish or English strings in widgets. The app ships with two locales: English (en) and Turkish (tr).

> **Contract reference:** strings coming FROM the API (errors, notifications, enum values) are resolved via the envelope defined in `api-agent/children/user-facing-strings.md`. That file is canonical; this file covers the Flutter side — ARB setup, widget patterns, ICU syntax, locale switching.

## API Message Resolution

Strings the backend sends are keys, not final text. Resolve them through `AppLocalizations` (ICU-aware) and fall back gracefully.

### ProblemDetails envelope

Error responses carry `extensions.messageKey`, `extensions.placeholders`, and `extensions.fallback`. Resolve in this order:

```dart
// lib/core/localization/message_resolver.dart
extension MessageResolver on BuildContext {
  String resolveApiMessage({
    required String messageKey,
    Map<String, Object?>? placeholders,
    String? fallback,
  }) {
    // 1. Try current locale
    final localized = AppLocalizations.of(this).tryLookup(messageKey, placeholders);
    if (localized != null) return localized;

    // 2. Try English fallback (if not already in English)
    if (AppLocalizations.of(this).localeName != 'en') {
      final english = AppLocalizationsEn().tryLookup(messageKey, placeholders);
      if (english != null) return english;
    }

    // 3. Server-provided fallback
    if (fallback != null && fallback.isNotEmpty) return fallback;

    // 4. Last resort: return the key so developers see the gap
    debugPrint('i18n: no translation for key "$messageKey"');
    return messageKey;
  }
}
```

`tryLookup` is a small extension on the generated `AppLocalizations` class that does a reflective key lookup; if that's too hacky for your taste, generate a single `Map<String, String Function(Map<String, Object?>)>` from the ARB files at build time and use that instead.

### Validation errors (multiple)

ProblemDetails with `extensions.errors` is a list of `{ field, messageKey, placeholders, fallback }`. Map each one to the form field it belongs to:

```dart
final errors = problemDetails.extensions['errors'] as List;
for (final e in errors) {
  formState.setError(
    e['field'] as String,
    context.resolveApiMessage(
      messageKey: e['messageKey'] as String,
      placeholders: (e['placeholders'] as Map?)?.cast<String, Object?>(),
      fallback: e['fallback'] as String?,
    ),
  );
}
```

### Socket notifications

The payload carries `messageKey + placeholders + fallback`. Resolve the same way and render the resulting string (snackbar, toast, in-app banner).

### Enum values

Backend returns `{ "status": "completed" }`. Never display the raw value; derive the key:

```dart
extension EnumLocalization on BuildContext {
  String enumLabel(String entity, String field, String value) {
    final key = '${entity}_${field}_$value';
    return resolveApiMessage(messageKey: key);
  }
}

// usage
Text(context.enumLabel('task', 'status', task.status));
// lookup: task_status_completed → "Completed" (and the equivalent in every supported locale)
```

### Email / push notifications — NOT the UI's problem

Emails and push notifications are rendered on the backend (MailSender / PushDispatcher) from per-locale templates. The UI never resolves their text. If a push notification lands in the app's foreground handler, its `title`/`body` are already localized strings — just display them.

## l10n.yaml Configuration

This file lives at the project root and configures the `flutter gen-l10n` tool:

```yaml
# l10n.yaml
arb-dir: lib/l10n
template-arb-file: app_en.arb
output-localization-file: app_localizations.dart
output-class: AppLocalizations
output-dir: lib/l10n/generated
nullable-getter: false
```

> Do NOT add `synthetic-package: false`. The `synthetic-package` option was deprecated in Flutter 3.24 and removed in newer versions — generation always happens in the app package now. Including it produces a deprecation warning on every build.

> Do NOT create `l10n.yaml` before it's used. If `lib/l10n/*.arb` files don't exist and `MaterialApp` isn't wired with `AppLocalizations.localizationsDelegates`, the file is an orphan that just emits warnings. Add `l10n.yaml` when you add the first ARB file — not during initial scaffolding.

## ARB File Format

ARB (Application Resource Bundle) files are JSON with metadata. Each locale gets its own file.

### Template File (English)

```json
// lib/l10n/app_en.arb
{
  "@@locale": "en",

  "app_name": "ExampleApp",
  "@app_name": {
    "description": "The application name"
  },

  "nav_home": "Home",
  "nav_tasks": "Tasks",
  "nav_stats": "Stats",
  "nav_profile": "Profile",

  "common_retry": "Retry",
  "common_cancel": "Cancel",
  "common_save": "Save",
  "common_delete": "Delete",
  "common_edit": "Edit",
  "common_loading": "Loading...",
  "common_error_generic": "Something went wrong. Please try again.",
  "common_error_network": "No internet connection.",

  "login_title": "Welcome back",
  "login_email_label": "Email",
  "login_email_hint": "Enter your email",
  "login_password_label": "Password",
  "login_button_submit": "Log in",
  "login_link_forgot_password": "Forgot password?",
  "login_link_register": "Don't have an account? Sign up",

  "tasks_title": "My Tasks",
  "tasks_empty_title": "No tasks yet",
  "tasks_empty_subtitle": "Start your first task by tapping the + button",
  "tasks_count": "{count, plural, =0{No tasks} =1{1 task} other{{count} tasks}}",
  "@tasks_count": {
    "description": "Number of tasks with plural support",

        "type": "int",

    "description": "Task duration in minutes",

      "minutes": {
```

### Secondary Locale (e.g., `app_tr.arb`)

Each non-default locale ARB mirrors the template's exact key shape — only the values differ. Example skeleton:

```json
// lib/l10n/app_<locale>.arb
{
  "@@locale": "<locale-code>",

  "app_name": "ExampleApp",

  "nav_home": "...",
  "nav_tasks": "...",
  "nav_stats": "...",
  "nav_profile": "...",

  "common_retry": "...",
  "common_cancel": "...",
  // …all template keys, with translated values
}
```

Code generation will fail if a key from `app_en.arb` is missing here — that's the safety net for "forgot to translate."

## Adding New Strings

### Step-by-Step

1. Add the key + `@key` metadata to `app_en.arb`
2. Add the translation to `app_tr.arb`
3. Run code generation (happens automatically with `flutter gen-l10n` or on build)
4. Use `context.l10n.key` in the widget

### String Naming Convention

Format: `{feature}_{element}_{variant}` or `{feature}_{action}_{element}`

| Pattern | Example | Usage |
|---------|---------|-------|
| `{feature}_title` | `tasks_title` | Screen title |
| `{feature}_empty_title` | `tasks_empty_title` | Empty state heading |
| `{feature}_empty_subtitle` | `tasks_empty_subtitle` | Empty state description |
| `{feature}_button_{action}` | `login_button_submit` | Button label |
| `{feature}_label_{field}` | `login_email_label` | Form field label |
| `{feature}_hint_{field}` | `login_email_hint` | Form field hint |
| `{feature}_link_{action}` | `login_link_register` | Clickable text |
| `{feature}_error_{type}` | `tasks_error_load` | Error message |
| `common_{action}` | `common_retry` | Shared across features |
| `nav_{tab}` | `nav_home` | Bottom navigation |

## Plurals

ICU message format for plural strings:

```json
"tasks_count": "{count, plural, =0{No tasks} =1{1 task} other{{count} tasks}}",
"@tasks_count": {
  "placeholders": {
    "count": { "type": "int" }
  }
}
```

Usage:
```dart
Text(context.l10n.tasks_count(tasks.length))
```

## context.l10n Extension

A shortcut to access localized strings without the verbose `AppLocalizations.of(context)!`:

```dart
// lib/core/extensions/context_extensions.dart
import 'package:flutter/material.dart';
import 'package:example_app/l10n/generated/app_localizations.dart';

extension LocalizationExtension on BuildContext {
  AppLocalizations get l10n => AppLocalizations.of(this);
}
```

Usage:
```dart
// Instead of
AppLocalizations.of(context)!.tasks_title

// Use
context.l10n.tasks_title
```

## Locale Switching at Runtime

Users can change the language in settings. The selection is persisted with SharedPreferences.

```dart
// lib/core/providers/locale_provider.dart
import 'dart:ui';
import 'package:flutter_riverpod/flutter_riverpod.dart';
import 'package:shared_preferences/shared_preferences.dart';

final localeProvider = StateNotifierProvider<LocaleNotifier, Locale?>((ref) {
  return LocaleNotifier(ref);
});

class LocaleNotifier extends StateNotifier<Locale?> {
  LocaleNotifier(this._ref) : super(null) {
    _loadSavedLocale();
  }

  final Ref _ref;
  static const _key = 'app_locale';

  Future<void> _loadSavedLocale() async {
    final prefs = await SharedPreferences.getInstance();
    final code = prefs.getString(_key);
    if (code != null) {
      state = Locale(code);
    }
  }

  Future<void> setLocale(Locale locale) async {
    state = locale;
    final prefs = await SharedPreferences.getInstance();
    await prefs.setString(_key, locale.languageCode);
  }

  Future<void> clearLocale() async {
    state = null; // Falls back to system locale
    final prefs = await SharedPreferences.getInstance();
    await prefs.remove(_key);
  }
}
```

### Language Picker Widget

```dart
class LanguagePicker extends ConsumerWidget {
  const LanguagePicker({super.key});

  static const _supportedLocales = [
    (locale: Locale('en'), label: 'English'),
    (locale: Locale('tr'), label: 'Turkish'),
  ];

  @override
  Widget build(BuildContext context, WidgetRef ref) {
    final currentLocale = ref.watch(localeProvider);

    return Column(
      crossAxisAlignment: CrossAxisAlignment.start,
      children: [
        Text(
          context.l10n.settings_language,
          style: context.text.titleMedium,
        ),
        const SizedBox(height: 8),
        ..._supportedLocales.map(
          (entry) => RadioListTile<Locale>(
            title: Text(entry.label),
            value: entry.locale,
            groupValue: currentLocale,
            onChanged: (locale) {
              if (locale != null) {
                ref.read(localeProvider.notifier).setLocale(locale);
              }
            },
          ),
        ),
        RadioListTile<Locale?>(
          title: Text('System'),
          value: null,
          groupValue: currentLocale,
          onChanged: (_) {
            ref.read(localeProvider.notifier).clearLocale();
          },
        ),
      ],
    );
  }
}
```

## Date and Number Formatting

Use `intl` package for locale-aware formatting. Do NOT put format strings in ARB files.

```dart
import 'package:intl/intl.dart';

// Date formatting
final dateFormatter = DateFormat.yMMMd(locale.languageCode);
Text(dateFormatter.format(task.startedAt)); // locale-aware: "Apr 14, 2026", etc.

// Number formatting
final numberFormatter = NumberFormat.decimalPattern(locale.languageCode);
Text(numberFormatter.format(task.distanceKm)); // locale-aware: "3.5" / "3,5"

// Currency formatting
final currencyFormatter = NumberFormat.simpleCurrency(locale: locale.languageCode);
Text(currencyFormatter.format(price)); // locale-aware currency symbol + grouping
```

## Anti-Patterns

### 1. Hardcoded strings
```dart
// BAD
Text('My Tasks')
// GOOD
Text(context.l10n.tasks_title)
```

### 2. String concatenation for messages
```dart
// BAD
Text('Hello, ' + user.name + '! You have ' + count.toString() + ' tasks.')
// GOOD
Text(context.l10n.task_greeting(user.name))
Text(context.l10n.tasks_count(count))
```

### 3. Missing @metadata in ARB
```dart
// BAD: no description, no placeholder metadata
{ "task_greeting": "Hello, {name}!" }

// GOOD: full metadata
{
  "task_greeting": "Hello, {name}!",
  "@task_greeting": {
    "description": "Greeting with user name",
    "placeholders": {
      "name": { "type": "String", "example": "John" }
    }
  }
}
```

### 4. Using enum.name directly as user-facing text
```dart
// BAD
Text(task.status.name) // "completed" in all languages

// GOOD
Text(context.l10n.taskStatus(task.status.name))
// with ARB: "taskStatus_{value}" entries for each status
```

### 5. Only adding the template locale, forgetting the others
```dart
// ALWAYS add the new key to every supported `app_<locale>.arb`
// The build will fail if a key is in the template but missing from any translation
```
